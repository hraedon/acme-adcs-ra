## Why this exists

v1.5 shipped feature-complete automated CA-side revocation, live-reproven
end-to-end. A post-release review (2026-07-23) surfaced four gaps that qualify
the "done" state — three are validation/hardening, one (E-1) qualifies the
project's **cardinal** security claim ("blast radius bounded to the
`ACME-ServerAuth` template"). This plan closes them, in priority order.

The gaps, plainly:

1. **The bound is proven on the revocation side but has a hole on the enrollment
   side** (Finding E-1) — and the hole comes from an ADCS *default*, so most
   estates will have it.
2. **The recommended (two-identity) topology was never proven live** — only the
   weaker, opt-in single-identity path was.
3. **The PowerShell — where revocation correctness actually lives — has zero
   automated coverage;** all six v1.5 hardening defects hid there.
4. **The ADCS integration is only point-in-time proven** — cloud CI can't reach a
   CA, so "green CI" does not mean "the enrollment/revocation legs still work."

## Work items

### WI-035 — Finding E-1 remediation: bound the enrollment gMSA to `ACME-ServerAuth` *(the load-bearing one)*

**Context** (`docs/revocation-scope-validation.md` Finding E-1): the enrollment
gMSA has `primaryGroupID=515` (Domain Computers); the `Machine` template grants
`Enroll` to Domain Computers, so the gMSA can enroll `Machine` (clientAuth-capable)
certificates directly against `/certsrv/`, **bypassing** the RA's serverAuth-only
policy. WI-026 bounds the *RA* path, but a compromised gMSA that skips the RA is
not ACL-bounded to one template — so the cardinal guarantee holds only as long as
the attacker goes through the RA.

- Remediate the ambient over-privilege via one of (owner picks per estate; document
  the choice):
  - **(a)** set the gMSA's `primaryGroupID` to a group that confers no
    template-enroll rights;
  - **(b)** scope the `Machine` (and any other broad) template enrollment ACLs so
    Domain Computers membership does not confer `Enroll` on this account;
  - **(c)** remove the gMSA from the Domain Computers enrollment path via a
    dedicated group.
- Re-run the **enrollment-side** blast-radius check: as the gMSA, normal
  `ACME-ServerAuth` issuance still succeeds, **and** a `Machine`-template enroll is
  now **denied**. (The WI-030 harness already establishes the technique.)
- Update `docs/revocation-scope-validation.md`: flip E-1 from
  **UNRESOLVED** to **remediated + verified**, with the observed denial evidence
  (placeholders only — the identifier gate applies).
- **AC:** live proof that the gMSA can enroll `ACME-ServerAuth` (issuance
  unaffected) and **cannot** enroll `Machine`; E-1 section updated; no regression
  in the WI-028 issuance re-proof.

### WI-036 — Live-prove the recommended two-identity topology

**Context:** v1.5 documents the dedicated-revoker (two-identity) design as the
**recommended default**, but only the single-identity path was live-proven — the
`gMSA-acme-revoker$` identity does not exist in the lab. The recommended path
currently rests on "same mechanism, identity-agnostic" reasoning, not a live proof.

- Provision a dedicated `gMSA-acme-revoker$` on a **utility host** (not the CA, not
  the RA host): install the gMSA, grant Certificate-Manager (`0x2`, **not**
  Manage-CA `0x1`) on the CA security descriptor, apply the template-scoped
  `OfficerRights` (using the fixed `Set-OfficerRights.ps1`), add
  `Certificate Service DCOM Access`, and confirm it belongs to **no** broader
  certificate-manager group (the union-semantics constraint).
- Run the automated revocation round-trip through the revoker on the utility host
  (**without** `-LocalMode`): RA `revokeCert` → agent pull → CA revoke → RA confirm,
  with the WI-022 requester check and the WI-025 officer restriction both active.
- Assert **compromise independence**: confirm the enrollment gMSA holds **no**
  officer rights (`GetMyRoles` on the enrollment gMSA shows Enroll only), so the two
  capabilities fail independently — the property the two-identity design exists to
  buy.
- Record the two-identity round-trip in `docs/pre-pilot-checklist.md` alongside the
  single-identity one.
- **AC:** two-identity round-trip proven live; enrollment gMSA confirmed to hold no
  revoke capability; checklist updated.

### WI-037 — Pester test suite for the PowerShell *(highest-leverage; no lab required)*

**Context:** revocation correctness lives in PowerShell with zero automated
coverage; the v1.5 defects (certutil arg order, REG_SZ vs REG_BINARY, task-action
quoting, logon type, requester forwarding) were only caught by live lab work.
Cover the **pure-logic seams** — the parts that run under `pwsh` on Linux with no CA
— and mock the Windows-cmdlet boundaries.

Directly Linux-runnable (pure string/byte/regex logic):

- **`Set-OfficerRights.ps1` blob builder** — a **golden-bytes** regression test
  against a captured reference: owner SID `S-1-5-32-544`, callback ACE mask
  `0x00010000`, opaque `[SidCount][subject SIDs][template OID + UTF-16 null]`,
  all-subjects = `SidCount 1 + Everyone (S-1-1-0)`. This is the most delicate code
  in the repo (a byte-wrong blob is fail-closed on the live CA) and it has no
  regression net today. Add-ACE / remove-ACE round-trip on in-memory bytes.
- **`Register-MaintenanceTasks.ps1` action string** — `Build-SyncActionCommand`
  produces a single-quote-only `-Command` payload (no nested double quotes), routes
  the token via `$env:ACME_ADMIN_TOKEN` (never `-AdminToken`), and composes
  `-RequesterName` / `-DryRun` vs `-Execute` / `-PublishCrl` correctly.
- **`Revoke-Cert.ps1`** — reason-code validation (valid set; reason 7 rejected →
  exit 3), the `certutil` argument vector (`-config` precedes the `-revoke`/`-CRL`
  verb), and the requester-name comparison (case-insensitive; mismatch semantics).
- **`Sync-Revocations.ps1`** — pending-set parse, `-SkipPublishCrl` default vs
  `-PublishCrl`, exit-code contract (0 all-confirmed / 2 partial / 5 requester
  abort).

Windows-cmdlet-bound (mock or keep lab-only): `Get-TaskPrincipal`
(`New-ScheduledTaskPrincipal`), the registry-provider write in
`Set-OfficerRights.ps1`, `Invoke-RestMethod`. Where cheap, refactor the pure logic
into dot-sourceable functions so the test does not trigger a Windows-only
`Import-Module` at load time; mock the cmdlet calls otherwise.

- Wire a **`pwsh` + Pester** job into CI (runs on the Linux runner for the pure-logic
  tests — no Windows/CA needed).
- **AC:** Pester suite green in CI on Linux; covers the blob builder (golden bytes),
  the action-string builder, the reason/requester logic, and the exit-code contract;
  a documented note on which seams remain lab-only.

### WI-038 — Honest ADCS-integration decay control

**Context:** cloud CI cannot reach a CA and the monthly rot-canary only checks
Python dependencies, so the enrollment/revocation legs' "proven" state decays
silently between manual re-proofs.

- Formalize the existing ad-hoc re-proof (`wi015_reproof.py` + the officer/sync
  provisioning used in the v1.5 pass) into a **documented, repeatable live-reproof
  runbook** covering issuance + EKU + **both** revocation topologies, with scripted
  setup **and teardown** that returns the CA and RA to a pristine/parked baseline.
- Add a **cadence policy**: run the live re-proof at each release, before any pilot,
  and on a fixed interval (e.g. quarterly); record results in the checklist
  validation log. State plainly in `README.md` / `AGENTS.md` that **green CI does
  not mean the ADCS integration is verified**.
- *(Optional / future)* if a self-hosted Windows + ADCS runner ever exists, wire the
  re-proof as a manual-dispatch (`workflow_dispatch`) CI job.
- **AC:** a documented, scripted live-reproof runbook with clean teardown; a stated
  cadence; the decay caveat added to README/AGENTS.

### WI-039 *(minor)* — deterministic CI dependency install

**Context:** CI installs dev deps **unlocked** (`uv pip install -e ".[dev]"`); the
v1.5.0 release commit was red-lit by a `ruff` drift (0.15.17 → 0.15.19), fixed by
pinning `ruff`/`mypy`. Close the class rather than the instance.

- Either switch CI to `uv sync --locked` (respect `uv.lock` for **all** deps) or keep
  the linter pins and document the deliberate-bump procedure. Note: the whole sibling
  family uses the same `uv pip install -e ".[dev]"` pattern and is exposed to the same
  drift.
- **AC:** CI dependency resolution is deterministic; a linter/tool bump is a
  reviewed, intentional change.

## Sequencing

- **Lab-free first, in parallel:** WI-037 (Pester) and WI-039 (CI). WI-037 gives the
  regression net *before* any further PowerShell change lands.
- **One shared lab trip:** WI-035 (E-1) and WI-036 (two-identity) are both CA/AD
  provisioning work — do them together under a single setup/teardown (remediate E-1
  and verify the enrollment bound; create the revoker and prove the two-identity
  round-trip).
- **WI-038** formalizes the harness used in that lab trip — write the runbook *as*
  the WI-035/036 session runs, so it reflects the real steps.
- **Release v1.6.0** (version bump + tag + GitHub release) after all WIs land and the
  combined live re-proof passes.

## Explicitly out of scope

- Any new ACME feature surface or issuance-path behaviour change — this is hardening
  and validation only.
- In-band CRL write from the RA (still deferred by design; see threat-model §E).
- Changing the recommended two-identity default or the opt-in single-identity option
  — both remain.
- Procuring self-hosted CI infrastructure (WI-038 notes it as optional/future, not a
  blocker).
