# Plan 005 — v1.5.0: feature-complete (automated revocation + self-enforced serverAuth)

**Status:** LANDED — all work items implemented and merged; WI-028 live E2E
re-proof PASSED on the deployed build (2026-07-23: base issuance + WI-026 EKU
check + automated revocation round-trip, requester + officer restriction active).
The v1.5.0 release tag (WI-029) is the one remaining step. Authored 2026-07-23
by Claude (Opus 4.8). Successor to [Plan 004](004-automated-revocation-spike.md)
(WI-021 PASSED: template-scoped revocation proven live).
**Target version:** 1.5.0.

> **Strategic role.** v1.0 shipped a correct, live-proven RA whose one honest
> functional gap was **CA-side revocation** (out-of-band, manual) and whose one
> unenforced safety guarantee was **serverAuth-only** (rested entirely on ADCS
> template config). Plan 004 proved the revocation gap is closable without
> widening the enrollment gMSA's rights. **v1.5 = build that automation + make the
> serverAuth guarantee self-enforced.** After 1.5 the feature surface is complete;
> the remaining distance to production is operational (pilot checklist §B–E), not
> featural. Nothing here adds a signing key or grants the *enrollment* gMSA any
> CA-officer rights — the cardinal invariants hold.

## Ground truth (what already exists)

- **Revocation, RA side:** `revokeCert` records RA-store revocation + honest audit
  (`revocation_scope=ra-store-only`, `ca_crl_updated=false`) and surfaces the
  out-of-band step (`routes/revocation.py`, `revocation.py`). `RevocationLeg` is a
  clean protocol seam — a future in-band leg drops in without an ACME-surface
  change.
- **Operator revoke + reconcile:** `scripts/Revoke-Cert.ps1` (CA-officer runs
  `certutil -revoke`), `scripts/Reconcile-Revocation.ps1` + `reconcile_revocation.py`
  (read-only drift buckets, incl. the dangerous `revoked_in_ra_active_at_ca`).
- **Post-issuance verification:** `finalize.py::_issued_cert_san_violations`
  (MED-1) already inspects the *issued* cert and fails closed (audit +
  `server_internal`, no cert recorded) — the exact pattern WI-026 reuses.
- **Admin auth:** `routes/admin.py::_require_admin_token` (Bearer token from
  `ctx.config.admin_token`) — the pattern WI-023 reuses.
- **Proven config recipe (Plan 004):** the CA-enforced restriction is an
  `OfficerRights` callback-ACE SD — owner `S-1-5-32-544`, per-officer mask
  `0x00010000`, opaque `[SidCount][subject SIDs][template OID+null]`, all-subjects
  = SidCount 1 + Everyone. A validated builder exists (spike artifact).

## Theme A — Automated revocation (closes the functional gap)

### WI-022 — `Revoke-Cert.ps1` requester-check hardening  *(pure PowerShell; no issuance-path risk)*
- In `Confirm-SerialAtCa`, after confirming the serial exists at the CA, assert
  the CA-DB `Request.RequesterName` equals the enrollment gMSA. Add a
  `-RequesterName` param (default the gMSA's `DOMAIN\gMSA-acme-ra$`); refuse to
  revoke a serial not requested by it. This bounds any automation (or operator
  typo) to the RA's own issuance even before the CA-side officer restriction.
- Tests: extend the `Revoke-Cert.ps1` Pester/contract coverage — requester match
  → proceeds; mismatch → non-zero exit, no `certutil -revoke` called.
- **AC:** a serial issued by any other requester is refused with a distinct exit
  code; the gMSA's own serials still revoke.

### WI-023 — `GET /acme/admin/revocations/pending` (read-only pull endpoint)
- New route in `routes/admin.py`, guarded by `_require_admin_token` (reuse the
  existing Bearer pattern — the pull agent presents the admin token; no new auth
  mechanism). Emits the `revoked_in_ra_active_at_ca`-shaped set: for each cert the
  RA has marked revoked, `{serial, req_id, reason, revoked_at}` from the RA store.
- Read-only; no state change. This is the RA-side half of the closed loop (the
  agent already had `reconcile_revocation.py` for the DB-file path — the endpoint
  avoids sharing the SQLite file cross-host).
- Tests: unit (auth required; only ra-store-revoked/CA-active rows returned; empty
  when in sync).
- **AC:** authenticated GET returns the pending-CA-revocation set; unauth → 401.

### WI-024 — CA-side pull agent  *(the headline deliverable)*
- A scheduled task (runs on a **utility host**, not the CA — `certutil -revoke
  -config` is remote-capable and the agent runs as its own gMSA, so no
  Kerberos double-hop) under **`gMSA-acme-revoker$`**. Loop: `GET
  /acme/admin/revocations/pending` → for each serial, `Revoke-Cert.ps1` (which now
  self-checks requester, WI-022) → on success, POST back to a small RA admin
  callback (or re-run reconcile) so the RA audit flips `ca_crl_updated=true`.
- **Fail-visible, dry-run default:** first ships in report-only mode; a config
  flag arms it. Every auto-revoke lands in the CA DB (under the revoker identity)
  and the RA audit. One-directional (RA→CA only); the reverse bucket stays a human
  review item.
- **AC:** end-to-end, an RA `revokeCert` is reflected on the CA CRL by the agent
  within one cycle, with matching audit on both sides; dry-run makes no change.

### WI-025 — Revoker provisioning runbook + validated `OfficerRights` builder
- `docs/operations.md` ## Revocation runbook gains an **automated** section, and a
  new `scripts/Set-OfficerRights.ps1` productionizes the Plan-004 builder (owner
  SID + mask `0x00010000` + all-subjects=Everyone; a `Get-OfficerRights` reader
  for verify-by-readback). The runbook provisions `gMSA-acme-revoker$` with the
  **two hard constraints from the live spike**:
  1. **Not a member of any broader certificate-manager group** (officer rights are
     a union over the token — a broader-manager membership silently defeats the
     restriction).
  2. **Member of `Certificate Service DCOM Access`** (else revoke → `INVALID_DATA`).
  Plus: CA-Security `ManageCertificates` for the revoker, and `OfficerRights`
  scoping it to `ACME-ServerAuth` only. GUI path documented as the alternative.
- **AC:** following the runbook on a clean CA yields a revoker that revokes an
  ACME-ServerAuth cert and is denied any other template (`CERTSRV_E_RESTRICTEDOFFICER`)
  — the WI-021 result, reproducible by an operator.

## Theme B — Self-enforced serverAuth guarantee (closes the safety gap)

### WI-026 — EKU post-issuance verification  *(issuance-path change → earns a live re-proof)*
- Add `finalize.py::_issued_cert_eku_violations`, a sibling to
  `_issued_cert_san_violations`, called in `_finalize_complete` alongside MED-1.
  It requires the issued cert's Extended Key Usage to be **exactly `{serverAuth}`**
  (`1.3.6.1.5.5.7.3.1`): reject if EKU is **absent** (all-purpose), if it contains
  `clientAuth` (`…3.2`), `anyExtendedKeyUsage` (`2.5.29.37.0`), PKINIT, or **any**
  other OID. Fail closed the same way (audit `finalize-issued-cert-eku-mismatch`
  + `server_internal`, cert **not** recorded/served).
- **Why it matters:** the "blast radius bounded to spoofing internal TLS" property
  in `docs/threat-model.md` currently rests entirely on template config; a
  template that ever gained clientAuth/PKINIT would silently break it — the exact
  domain-takeover escalation the threat model names as the worst case short of a
  signing key. This makes the guarantee self-enforced by the RA, not assumed.
- Tests: fixtures for serverAuth-only (pass), no-EKU (reject), +clientAuth
  (reject), anyEKU (reject); mirror MED-1's test structure.
- **AC:** any issued cert whose EKU is not exactly serverAuth is rejected,
  audited, and never served — verified against real ADCS output in the re-proof.

### WI-030 — Revocation-scope validation: evidence document + thorough test matrix

**Deliverable:** `docs/revocation-scope-validation.md` — a durable, auditable
evidence report that proves the CA-enforced revocation scope **and its limits**,
with a reproducible harness. This is the officer-rights counterpart to the
`finalize.py` SAN/EKU tests: it turns *"a compromised revoker cannot revoke certs
outside the `ACME-ServerAuth` template"* into a reproduced, auditable claim rather
than an assertion — fitting the tool's audit/provenance charter and the pilot's
evidence needs. **Placeholders only** (no real CA name, serials, SIDs, template
OIDs — the public repo's identifier gate applies).

The document contains:
1. **Method & environment** — how a *clean* restricted officer is provisioned
   (per WI-025), how each case is executed (`certutil -revoke -config` run **as the
   officer** via the scheduled-task-as-principal technique), and how outcomes are
   read (process exit code + the `ICertAdmin::RevokeCertificate` HRESULT).
2. **The `OfficerRights` recipe** — the reproducible blob spec proven in Plan 004
   (owner SID `S-1-5-32-544`; per-officer callback ACE mask `0x00010000`; opaque
   `[SidCount][subject SIDs][template OID + UTF-16 null]`; all-subjects =
   SidCount 1 + Everyone `S-1-1-0`; all-templates = Everyone entry, no template;
   deny = `DENIED_CALLBACK`) plus the two hard provisioning constraints.
3. **The test matrix with results** (below) — each row: scenario → expected →
   observed HRESULT → status.
4. **Interpretation guide** — the observed HRESULTs and what each means, so an
   operator/auditor reads results unambiguously:
   `0` = permitted; `0x80094009 CERTSRV_E_RESTRICTEDOFFICER` = template denial;
   `0x8007000d ERROR_INVALID_DATA` = officer lacks `Certificate Service DCOM
   Access`; `0x80070057 ERROR_INVALID_PARAMETER` = malformed/absent `OfficerRights`
   (fail-closed: officer ops break rather than silently unrestrict).

**Test matrix — be thorough; run every row.** `[P]` = proven in the Plan-004
spike; `[R]` = to run and record (do these when the WI-025 officer is provisioned,
so no separate lab trip):

*A. Core template enforcement*
- **A1** `[P]` allowed template (`ACME-ServerAuth`) → revoke **SUCCESS** (`0`).
- **A2** `[P]` disallowed template (`Machine`) → **`CERTSRV_E_RESTRICTEDOFFICER`**.
- **A3** `[R]` disallowed **`DomainController`** (highest-value target) → denied.
- **A4** `[R]` disallowed template of a third class (`OCSP` / code-signing /
  `CAExchange`) → denied — breadth across template types.
- **A5** `[R]` **before/after control**: the same officer with `OfficerRights`
  *absent* revokes the A2/A3 targets (SUCCESS), proving the later denial is caused
  by the restriction, not a broken officer.

*B. Operation coverage (the restriction must gate every officer op, not one reason)*
- **B1** `[P]` hold (reason 6) on allowed → SUCCESS; removeFromCRL (reason 8, the
  un-hold) on allowed → SUCCESS.
- **B2** `[R]` hold (6) *and* removeFromCRL (8) on a **disallowed** template →
  both denied — confirms the officer-rights access check precedes the reason
  handling, so a single code path gates all reason codes (document this so the
  destructive reasons need not be exercised against real certs).

*C. Adversarial / scope-escape (the real security question)*
- **C1** `[P]` officer *also* in an unrestricted manager group (union semantics)
  → revokes a disallowed template ⇒ **constraint: the revoker must belong to no
  broader certificate-manager group.**
- **C2** `[R]` restricted-to-`ACME-ServerAuth` officer revokes an
  `ACME-ServerAuth` cert requested by a **different** principal → **SUCCESS** at
  the CA ⇒ the CA boundary is **template-scoped, not requester-scoped**. This is
  exactly why the **WI-022 agent-side requester check** exists (defense-in-depth);
  the doc must state this so no one mistakes the CA restriction for RA-issuance
  scoping.
- **C3** `[R]` a **`DENIED_CALLBACK`** entry actually denies (the deny path
  enforces, not only allow-by-omission).
- **C4** `[R]` officer with **no** `ManageCertificates` on CA Security → cannot
  revoke anything (control: CA-Security grant and `OfficerRights` are distinct
  gates).

*D. Provisioning failure modes (must fail closed/visibly, never silently unrestricted)*
- **D1** `[P]` missing `Certificate Service DCOM Access` → `INVALID_DATA` (a
  *visible* failure, not a silent bypass).
- **D2** `[P]` malformed/absent `OfficerRights` → `INVALID_PARAMETER` for **all**
  officer ops (management breaks — fail-closed — rather than silently unrestricting).

*E. Durability*
- **E1** `[P]` restriction survives a `certsvc` restart.
- **E2** `[R]` restriction survives a CA host reboot.

**AC:** every `[R]` row is executed and recorded (expected == observed) in
`docs/revocation-scope-validation.md`; the doc reproduces the WI-021 allow/deny
result, both deployment constraints, and the interpretation guide; identifiers are
placeholders and the CI identifier gate passes.

## Sequencing

1. **WI-026 first** (small, independent, issuance-path) — land it early so it
   rides the *next* re-proof rather than forcing an extra one.
2. **WI-022** (PowerShell, no risk) — foundation for the agent.
3. **WI-025** (builder + runbook) — provisioning the revoker; validates the
   Plan-004 recipe as a shipped artifact.
4. **WI-023 → WI-024** — endpoint then agent (the biggest piece; dry-run first).
5. **WI-030** — run the validation matrix on the officer provisioned in WI-025
   (shared setup with WI-024/028) and write the evidence doc.
6. **WI-027** alongside; **WI-028** gates; **WI-029** ships.

WI-022/023/025/026 are independent and parallelizable; WI-024 depends on
023+025; WI-030 rides the WI-025/028 officer setup; the release (029) depends on
028 and on WI-030's evidence doc being complete.

## Explicitly out of scope for 1.5 (deliberate)

- **Granting the *enrollment* gMSA officer rights / an in-band `RevocationLeg`** —
  the concentration Plan 004 exists to avoid.
- **Bidirectional RA↔CA revocation reconciliation** (auto-reflecting CA-initiated
  revocations back into the RA) — the reverse bucket stays human-reviewed.
- **Mode C transport / ARI (RFC 9773) / account deactivation** — later, not gaps
  that block "feature complete."
- **Operator-owned pilot items (§B–E)** — deployment, tracked separately.
