## On the project

This is a genuinely well-engineered RA. The thing that stands out is that the
security discipline is *mechanical, not aspirational*: every order transition
is CAS-guarded, the audit store is unconditional, and the "no signing key"
cardinal rule is enforced by an architecture test with positive AND negative
controls (so it can't silently become a no-op). The threat model and
pre-pilot checklist are honest about the one real limitation (CA-side
revocation is out-of-band by design, to keep the gMSA least-privileged)
rather than hand-waving it. For a FastAPI/SQLite project in the issuance
path, that's rarer than it should be.

The shape that serves it best is the CAS-everywhere store + the
deterministic-policy chokepoint. The shape that's starting to creak is
`routes/acme.py` — it's a 786-line file doing directory, accounts, orders,
challenges, finalize orchestration, revocation, keyChange, and rate limiting.
Finalize was already decomposed into `finalize.py` (good); the rest hasn't
been. It's not broken, but it's the file a future agent will be most
uncomfortable editing. The docs-vs-code drift I spent half the session fixing
is the other creak: the narrative lagged the shipped state by two work items
because reflections/plans/README each drift independently with no single
source of truth.

## On the work done

The highest-value thing was MED-1, and it was *not* on any prior plan or
reflection — it came out of the security subagent reading the code fresh. The
RA validated SANs on the CSR but trusted the ADCS template to echo them onto
the cert. That's the core security control (SAN scope is the whole
authorization surface, since the challenge is a no-op), and it was only half-
enforced. The adversarial reviewer immediately found the obvious hole in my
first cut (I only checked DNS SANs; a template injecting an RFC822Name/IP SAN
would bypass it) — which is exactly why the adversarial pass exists on
issuance-path code. The final version rejects any non-DNS SAN and normalizes
trailing dots/case to match `policy.py`, so it won't false-reject the
lab-proven flow.

MED-2 (the `won_cas` signal) was the one explicitly-named "nice-to-have" from
the 2026-07-06 reflection, so that was satisfying to close — the route no
longer guesses about a lost CAS from timestamp inequality. LOW-4 (UNIQUE
index) is belt-and-suspenders, but the adversarial M-2 finding (CREATE UNIQUE
INDEX validates existing rows and would crash Store init on a dirty upgrade)
was a real footgun I'd have shipped without the review; the graceful-migration
form is the right call for infra that must not fail to start.

What I'm confident in: the MED-1/MED-2 logic and tests, the doc fixes (every
factual contradiction is resolved). What I'd want a second pair of eyes on:
nothing critical — but the MED-1 "subset, not exact equality" choice is a
deliberate tradeoff (a cert *missing* a requested SAN passes, since that's a
malfunction not a privilege escalation). If the project ever wants stricter
"the cert must exactly match the order," that's a one-line change and a test
update.

## Gaps to flag

- **`docs/pre-pilot-checklist.md:130-148`** — the WI-015 validation-log entry
  contains real identifiers (`mvmcitest01`, `ad-MVMCA01`, `Hraedon Root CA`,
  `wi015-reproof.ad.hraedon.com`, a real cert serial, `HRAENET\gMSA-acme-ra$`).
  These are **homelab/hraedon identifiers, which are ALLOWED** per the
  canonical denylist (only the work-domain El Rio set is forbidden) — so this
  is NOT a violation, and the identifier-gate CI (which keys off the secret)
  will confirm. Flagging only because the docs-coherence reviewer (lacking
  that context) flagged it as a hard-rule violation; the next agent should not
  "fix" it by redacting allowed lab identifiers.
- **MED-1 is gated to `serverAuth`-only templates by design but does not
  itself check the EKU.** If a future template adds clientAuth, MED-1 won't
  catch it — the SAN check is orthogonal to EKU. The architecture relies on
  the template being server-auth-only (a threat-model condition); MED-1
  doesn't strengthen that. A future hardening could verify the issued EKU too.
- **`src/acme_adcs_ra/store.py` LOW-4 index** is now created in a migration
  with try/except that logs + continues on a dirty upgrade. That means a
  deployed DB with duplicate `order_id` rows will run *without* the DB-level
  guarantee (still protected by the CAS). The log message names the
  remediation; make sure an operator would actually see that ERROR log.
- **No reflection exists for the Plan 003 / WI-015 session** (the work in
  commits 7d5c5b9 + fd29c81). This session's reflection picks up downstream
  but the "how did Plan 003 go" record is absent from reflections/.
- **LOW-2 (expiry guard in `_maybe_ready_order`)** is defense-in-depth with no
  issuance impact; the test proves the guard. But it means an expired pending
  order now stays `pending` until the sweep moves it to `invalid` — confirm
  the sweep cadence is short enough that this isn't a surprising long-lived
  `pending` in monitoring.
