## On the project

The core issuance codebase is unusually disciplined for a FastAPI/SQLite
project: every order transition is CAS-guarded, the audit store is
unconditional, JWS/EAB handling is careful, and the architecture test
mechanically enforces the "no signing key" cardinal rule. The documentation
is also strong — the threat model and pre-pilot checklist are honest about
what the RA cannot do (notably CA-side revocation) rather than pretending
away the gap.

The remaining distance to a responsible 1.0 was never more code features; it
was operator enablement. Plan 002 correctly identifies that, and this session
closed that gap. The result is a project where the code, docs, and checklist
are coherent: an operator can now pick up `docs/operations.md` and know what
to paste, register, and monitor.

## On the work done

The implementation subagent did the bulk of the writing and the unit tests,
which I then reviewed and corrected. The biggest correction was the EAB
helper output format: the first version used `ACME_RA_EAB_ALLOWLIST__0__KID`
style keys, which pydantic-settings actually parses as a dict, not a
`list[EABEntry]`. I rewrote `scripts/eab.py` to emit a JSON array for the
allowlist and added a test that loads the output into `RAConfig` via a real
env file. That kind of detail is exactly why "write the helper" is not
enough — it has to be loadable by the config the app actually uses.

The three security fixes feel correct:

- M-1 closes a real silent-failure path in the out-of-band revocation loop.
- M-2 removes the one non-CAS order transition in the challenge path.
- M-3 makes concurrent revocation idempotent and audit-safe.

I'm slightly less confident in the M-3 idempotency heuristic in the route
(`updated.revoked_at != revoked_at`). It relies on timestamp inequality, so
two revocations in the same sub-second window could still emit a duplicate
audit. The store-level CAS itself is sound; the duplicate-audit prevention is
best-effort. I think this is acceptable for 1.0 but worth a follow-up if the
project ever wants a stronger "won CAS" signal from the store.

## Gaps to flag

- `src/acme_adcs_ra/routes/acme.py:525-536` — M-3 idempotency detection uses
  `updated.revoked_at != revoked_at`; a same-second race could duplicate audit.
  Low likelihood, but the duplicate-audit guarantee is not strict.
- `scripts/Register-MaintenanceTasks.ps1` embeds the admin token in the task
  action at registration time. This is the standard Windows pattern, but
  rotating the token requires re-running the script. The runbook documents
  this; verify it in the lab.
- `docs/operations.md` has a lot of guidance but no executable retention
  script. That is intentionally operator-owned, but make sure the operator
  actually writes one before audit_log grows unbounded.
- The `tests/test_revocation.py` M-3 tests create a self-signed cert with
  `CertificateBuilder().sign()` for the 404 path. This is fine in tests but
  could surprise a future agent reading the architecture guard; it is scoped
  to tests only.
