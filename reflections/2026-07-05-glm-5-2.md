## On the project

The project's discipline around the issuance path continues to impress.
The separation of `RevocationLeg` protocol → fake/dev leg → production leg
made this change almost entirely local to `revocation.py` + the route's
audit/response shaping — the store, the state machine (CAS-guarded
`transition_to_revoked`), and the ACME-surface contract (200 on success,
H-1 410 on revoked cert, H-4 idempotent re-revoke) all stayed untouched.
That is the payoff of the decomposition work done in WI-001..005.

The threat model is unusually honest. Every gap has a status and a section
pointer, and the §E revocation decision (out-of-band, preserve least
privilege) is recorded with a date and a rationale — not a TODO. Updating
§E/§7/§8/checklist §F/AGENTS.md/plan 002 to all tell the same story took
care but was straightforward because the decision was already made. The
"no signing key, ever" invariant and the architecture test that enforces
it are the load-bearing guardrails; my changes touch neither.

## On the work done

**The Python side was the easy part.** `CertsrvRevocationLeg.revoke()` now
returns `RevocationResult(revoked=True, metadata={"revocation_scope":
"ra-store-only", "ca_crl_updated": "false", "serial": ...})` instead of
raising. The route reads the metadata, enriches the audit with
`revocation_scope`/`ca_crl_updated`/`req_id`, and returns 200 with an
`out_of_band_revocation` hint when `ca_crl_updated == "false"`. The
defaulting logic (`get(..., "ra-store-only")` / `get(..., "false")`)
defaults to the *most honest* (most conservative) value — a leg that
forgets to set metadata cannot accidentally claim the CRL was written.

**The PowerShell script was the hard part — and the adversarial review
earned its keep there.** Round 1 found 5 blocking issues: `certutil -revoke`
accepts serials only (not ReqIDs — the `-ReqID` path was fundamentally
broken), the `-view` filter was positional instead of `-restrict`, the
column name was wrong, `-publishCRL` is not a verb (should be `-CRL
republish`), and the confirmation step matched localized English header
strings. Round 2 found 2 more: `-notmatch` on a string array filters
(always truthy) instead of testing — the confirmation step was
non-functional; and `Write-Error` + `$ErrorActionPreference = "Stop"`
swallowed every custom exit code. All fixed.

**What I'm most confident in:** the audit-honesty invariant. Every
`certificate-revoked` audit event now carries `revocation_scope` and
`ca_crl_updated`; the failed path (leg raises) carries neither (a failure
cannot honestly claim a scope); the idempotent already-revoked path
short-circuits before the leg. The `TestOutOfBandRevocation` +
`TestFakeLegRevocationResponseShape` + the failed-path test cover all
three branches.

**What I'm least confident in:** the `Revoke-Cert.ps1` script is
unit-testable only on a Windows host with a live CA — the AC requires lab
validation revoking a throwaway cert, which is part of the WI-015 live
re-proof gate. The certutil syntax is now correct per the docs and the
repo's own `lab/spike_mode_a.py` pattern, but I have not run it against a
real CA. The locale-independent hex-serial confirmation (`[regex]::Escape`
on the joined output) is sound in principle but unvalidated against real
non-English ADCS output.
