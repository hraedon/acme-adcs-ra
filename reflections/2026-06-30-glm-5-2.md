(2026-06-20), the codebase is well-structured (decomposed routes, CAS-guarded

What stands out is the discipline around the issuance path. Every change to the
enrollment leg earns an adversarial review and, where it touches issuance
behavior, a live lab re-proof. The separation of `EnrollmentDenied` (CA policy
denial → 400, CAS-revert to ready) from `EnrollmentTransportError` (transport
failure → 503, retry) is the load-bearing state machine distinction — getting
the disposition classification right (WI-007) directly affects whether a
client gets a hard failure or polls indefinitely.

## On the work done

**WI-009** was the smallest and highest-confidence change — reuse
`validate_dns_name` at order creation. The interesting part was updating the
two existing wildcard CSR tests: they previously created orders with wildcard
identifiers (`*.WORK-DOMAIN.local`), which WI-009 now rejects at newOrder. The
fix was to use a valid order identifier and keep the wildcard in the CSR — the
CSR gate (`_reject_wildcard_sans`) fires before the CSR⊆order check, so the
tests still correctly assert `badCSR`. The adversarial review caught a missing
>253-char test and a less-specific wildcard error message, both addressed.

**WI-006** was documentation-only — no behavior change. The adversarial review
caught a consistency nit: the pre-existing inline comment ("stable for the
host") read as overconfident next to the new docstring that qualifies the
assumption. Fixed by softening the comment. A cross-reference was also added
to `certsrv-setup.md` Mode C section.

**WI-007** was the most involved. I spent significant time researching the
actual `certfnsh.asp` response format — the reference implementation
(magnuswatn/certsrv) also uses English locale strings, and the only truly
locale-independent signal is the `certnew.cer?ReqID=<n>&` success URL. The key
insight: the disposition message is in quotes regardless of locale, and a
word+space prefix before the quote distinguishes it from HTML attributes
(`href="..."`). The adversarial review caught two real issues: (1) JavaScript
`return "..."` patterns could false-positive as denied — fixed by stripping
`<script>` blocks before the denied regex; (2) a denied page with `ReqID=` but
no matchable quoted message would be misclassified as pending — documented as a
known limitation with a test. The denied check intentionally runs BEFORE the
pending check so a denied page with both signals is correctly classified.

The part I'm least confident in: the `\b\w+\s+"([^"]{5,})"` denied regex is a
heuristic, not a structural parse. It works for the test fixtures (English and
German), but real ADCS pages in other locales might use colon separators
(`Disposition: "..."`) or HTML entities (`&quot;...&quot;`) that the regex
misses. The live re-proof required by the WI-007 AC should capture real
`certfnsh.asp` response bodies to validate the heuristic. The "unknown"
disposition with a body snippet is the safety net — it's fail-loud, not
fail-silent.

## Gaps to flag

- **WI-007 denied regex is a heuristic, not a structural parse**
  (`enrollment.py:392`): the `\b\w+\s+"([^"]{5,})"` pattern matches
  word+space+quoted-string, which works for "is '...'" / "lautet '...'" but
  misses colon-separated dispositions (`Disposition: "..."`) and HTML-entity
  quotes (`&quot;...&quot;`). The live re-proof should validate against real
  non-English ADCS pages. If the heuristic fails, the "unknown" disposition
  surfaces the body snippet (fail-loud).

- **Regista migration pending** (`acme_adcs_ra` schema missing migration 35):
  `agent-notes breadcrumb reconcile --apply` failed because of a pending
  regista migration. WI-006, WI-007, WI-009 are resolved in git (commit
  `2c3d81f`) but not yet transitioned to `closed` in the work-item DB. The next
  `reconcile --apply` after the migration will close them.

- **WI-007 misclassification risk for denied-as-pending**: if a denied page
  has `ReqID=` but no matchable quoted message, it's classified as pending
  (EnrollmentTransportError → 503 retry, order stays `processing`). The client
  polls until order expiry instead of getting a 400. This is fail-safe (no
  cert issued) but operationally noisy (stuck `processing` order needs manual
  reclaim). The error message hedges ("manager approval may be on, or the
  template policy denied the request") to signal the ambiguity.

- **`negotiate_auth._server_cert_der()` side-channel probe** (WI-006): the
  CBT is derived from a separate TLS connection, not the enrollment HTTP
  connection. This is now documented but not fixed. If a multi-backend
  deployment is ever chosen, this becomes a real WI — the fix would be a
  custom `HTTPAdapter` that extracts the cert from the auth connection.
